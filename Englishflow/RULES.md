# RULES

1. 默认每次完成一个TODO。

如果用户明确要求完成一个Milestone，则允许一次完成该Milestone内所有TODO。

完成整个Milestone后立即停止。

2. 完成当前开发范围后立即停止。

开发范围由用户指定：

- 一个TODO
或
- 一个Milestone

不要自行扩大范围。

3. 完成我指定的开发范围后立即停止，等待我的确认。

开发范围可能是：
- 一个TODO
- 一个Milestone

不要自行继续开发后续内容。

4. 直接修改项目文件，不要只给示例代码。

5. 不要删除已有功能。

6. 不要为了优化而重构整个项目。

7. 优先复用已有代码。

8. 如果预计修改超过5个文件或300行代码，请先提出拆分方案，并等待确认。

9. 如果发现更好的方案，请提出建议，但不要直接修改。

10. 每次开始开发前，请先阅读：

* PROJECT.md
* TODO.md
* RULES.md

11. 完成任务后，请更新TODO.md和CHANGELOG.md。

12. 只解释设计原因，不解释基础语法。

13. 解释控制在200字以内。

14. 如果需求与PROJECT.md冲突，以PROJECT.md为准。

15. 如果任务超出当前Milestone，请提醒我，不要提前开发。

16. 项目目标是长期维护，而不是一次生成全部代码。

17. 尽量减少Token消耗，优先采用最简单可靠的方案。

18. 如果可以拆成更小的TODO，请主动建议拆分。

19. 如果无法确定需求，请先提问，不要自行猜测实现方式。

20. 不要修改与当前TODO无关的文件。

21. 默认输出最少必要内容，不要重复PROJECT、TODO或已解释过的内容。

22.

如果用户指定：

完成Milestone，

则：

允许自动执行该Milestone所有TODO，

执行过程中不要反复等待确认，

仅在：

①预计修改超过300行

②需要重大架构调整

③发现PROJECT冲突

时暂停。
